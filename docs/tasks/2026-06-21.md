- 신규 Redis·Docker 운영 변경에 대한 당일 작업 선등록, 변경 영향 기록 및 장기 기술 문서 반영이 없다.

### 재구현 재점검 결과
- 기존 `IndentationError`는 해소되어 수정 Python 17개 파일의 개별 컴파일과 `app.main` import가 통과했다.
- `is_running` 수동 상태 계약은 미완료다. `order_reconciler._resume_user_if_safe()`와 주문 발견 경로가 종결 시 `is_running=True`를 자동 기록하고, 강제 청산도 실행 전후 `False/True`로 변경한다. 정지 API·관리자·텔레그램은 구현이 없는 `disable_auto_resume_for_user()`를 계속 호출한다.
- KIS 주문 의도를 Redis 락 획득 전에 `SUBMITTING`으로 커밋한다. 동일 종목 락 획득에 실패한 요청은 증권사에 전송되지 않았지만 미해결 주문으로 영구 원장에 남아 다음 주기와 사용자 전체 거래를 차단할 수 있다.
- 검토 도중 매도 실행의 예외 경로가 다시 수정되어 현재는 락 해제 `finally`를 통과한다. 다만 매수 예약 실패 경로는 여전히 락 해제 구문 밖에서 반환한다.
- `VirtualBalance.reserve()`는 실제 가용 현금과 비교하지 않고 예약 총액만 증가시킨 뒤 항상 `True`를 반환하므로 초과 사용 방지 기능을 수행하지 않는다. 강제 청산의 사용자 단위 `clear()`는 다른 실행의 예약 소유권을 구분하지 않는다.
- 로컬 가상환경에는 선언되지 않은 `redis 8.0.0`이 설치돼 있지만 `requirements.txt`, `Settings.REDIS_URL`, 환경 예시, Redis 종료 훅이 없다. Redis 6379 리스너도 없어 재현 가능한 실행 환경이 아니다.
- 테스트의 자동 Redis 대역은 스케줄러 참조를 패치하도록 재수정됐지만, 슬롯 할당과 강제 청산이 함수 내부에서 `VirtualBalance`를 다시 import하여 대역을 우회한다. 실제 Redis 경로가 실행되어 `Settings.REDIS_URL` 부재로 거래 흐름·강제 청산 테스트가 실패한다.
- `broker_orders.resume_after_resolution` 제거 Alembic 마이그레이션이 없어 `alembic check`가 계속 실패한다.
- KIS 전량 청산의 주문 의도에 보유종목 `strategy_type`이 여전히 전달되지 않아 기본 전략 외 체결 반영 위험이 남아 있다.
- 작업 과정의 일회성 패치 스크립트 `patch_is_running.py`, `patch_locks.py`, `rewrite_inner_funcs.py`가 백엔드 루트에 미추적 파일로 남아 있다.
