### 변경 영향 기록
- 변경 분류: `운영 절차`, `로컬 개발 기동 흐름`
- 생산자: 루트 `start_dev.bat`의 프론트엔드 프로세스 호출 및 서버 준비 상태 판정
- 소비자: Windows 개발자, `frontend/package.json`의 `local` 스크립트, `backend/run.py local`
- 데이터 경계: 사용자 데이터와 API 데이터는 변경하지 않으며 로컬 `cmd.exe` 프로세스와 8000·3000 포트 준비 상태만 확인
- API 계약: HTTP 요청·응답은 변경하지 않고 백엔드 8000과 프론트엔드 3000이 실제 수신 가능한 상태를 기동 성공 조건으로 강화
- 문서 반영: `README.md`의 기존 원클릭 실행 설명과 동일한 동작을 복구하므로 별도 장기 문서 변경 없음
- 회귀 테스트: 프론트엔드 직접 실행, `start_dev.bat --check`, 배치 호출 실패 시 창 유지, 서버 준비 상태 판정 검증
- 미해결 위험: 실제 통합 기동 검증 중 로컬 개발 서버 프로세스를 일시적으로 실행하고 검증 후 종료해야 함
- 인수인계: `start_dev.bat`의 `start` 명령과 포스트 스타트 대기 루프부터 확인

### 원인 및 수정 결과
- 원인 1: 3000 포트 미사용 검사에서 남은 `ERRORLEVEL=1`을 `start` 성공 직후 다시 검사해 백엔드만 연 상태에서 프론트엔드 실행 전에 실패 처리함
- 원인 2: `npm.cmd`를 `call` 없이 실행해 npm 실패 시 프론트엔드 터미널이 닫히고 오류 확인이 불가능했음
- 수정: `start` 반환값 대신 실제 포트 준비 상태로 성공을 판정하고, `call npm.cmd run local`로 프론트엔드 셸을 유지함
- 수정: 한쪽 StockAuto 서버가 이미 실행 중이면 해당 서버를 재사용하고 꺼진 서버만 시작하도록 변경함
- 수정: 표준 입력이 없는 환경에서 실패하는 `timeout`을 PowerShell `Start-Sleep`으로 교체함
