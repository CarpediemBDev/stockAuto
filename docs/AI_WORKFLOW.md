이 문서는 작업자가 바뀌어도 변경 의도, 데이터 경계, 검증 상태가 끊기지 않도록 하는 실행 절차의 SSOT입니다. 최상위 권한은 프로젝트 루트의 `AGENTS.md`에 있으며, 이 문서는 그 규칙을 실제 작업 형식으로 구체화합니다.

- 이 문서는 StockAuto 전용 적용본입니다. StockAuto의 경로, 도메인 구조, 하네스, 거래 모드, 브로커, 금융 계산, 스캐너 데이터 경계만 이 문서에 둡니다.
- StockAuto 작업 중에는 `C:\Users\Im\.codex\skills` 아래 개인 설치본을 원본으로 읽거나 수정하지 않습니다.
- 새 프로젝트를 만들 때는 `skills/common-ai-workflow`를 복사해 공통 원칙을 먼저 적용하고, 이 문서의 StockAuto 전용 경로와 명령어를 그대로 복사하지 않습니다.

## 1. 문서 우선순위

충돌이 있으면 아래 순서로 판단합니다.

1. 런타임 system/developer 지침
2. `skills/common-ai-workflow/SKILL.md`: 모든 프로젝트에 재사용할 공통 AI 작업 원칙
3. `AGENTS.md`: StockAuto 절대 규칙과 승인·Git 통제
4. `docs/tasks/YYYY-MM-DD.md`: 당일 작업 일지, 현재 작업 상태와 사용자 승인 여부
5. 연결된 GitHub Issue 또는 Linear 이슈: GitHub는 버그 기술 원장, Linear는 중요 장기 작업과 우선순위
6. 도메인 계약 문서: 데이터 흐름, API, 스키마, 전략 규칙
7. 실제 코드와 테스트: 현재 구현 사실

문서와 코드가 다르면 코드를 그대로 정답으로 간주하지 않습니다. 차이를 현황판과 연결 이슈에 기록하고 어떤 계약이 맞는지 확인한 뒤 함께 정렬합니다.

## 2. 새 AI 시작 절차

1. 루트 `AGENTS.md`를 전체 확인합니다.
2. `docs/tasks/`에서 최신 일자 파일과 최근 `[/]`, `[ ]` 항목을 확인합니다.
3. 작업 항목에 연결된 GitHub/Linear 이슈를 확인합니다. 도구가 없으면 동기화됐다고 가정하지 않습니다.
4. 조회용 `git status`로 기존 사용자 변경과 미추적 파일을 확인합니다.
5. 변경할 파일, 호출자, 소비자, 테스트, 관련 문서를 실제 검색으로 확인합니다.
6. 신규 작업을 당일 현황판에 `[ ]`로 선등록하고 실제 시작 시 `[/]`로 전환합니다.
7. 아래 `변경 영향 기록`을 먼저 작성한 뒤 구현합니다.

신규 기능 구현 또는 기존 기능 확장 시에는 구현 전에 동일·유사 기능을 먼저 검색합니다. 같은 검증, DB 변경, API 호출, 캐시 갱신, 토스트 처리, 수수료·손익·수량 계산이 이미 있으면 새로 복사하지 말고 기존 원장 모듈을 확장하거나 원장 모듈을 새로 정한 뒤 모든 소비자가 그곳을 호출하게 합니다.

## 3. 계약 영향 변경 판정

다음 중 하나라도 바뀌면 계약 영향 변경입니다.

- API 경로, HTTP 메서드, 인증, 권한, 요청·응답 필드
- 사용자별 데이터와 공용 데이터의 분리 또는 재결합
- 전역 캐시, 캐시 키, 무효화, 백그라운드 갱신 방식
- 스케줄러 주기, 중복 실행 가드, 재시도·재개 조건
- DB 모델, 마이그레이션, 기본값, 데이터 보존 정책
- 외부 증권사·데이터 공급자 계약 또는 실패 폴백
- 프론트엔드가 의존하는 상태·폴링·에러 처리 계약
- 검증 하네스, CI, 배포 또는 운영 절차

## 4. 변경 영향 기록 형식

계약 영향 변경 시 당일 현황판에 아래 블록을 그대로 사용합니다. 각 값은 실제 파일·함수·API·테스트 이름으로 작성하며 `없음`, `해당 없음`만 적지 않습니다.

```markdown
### 변경 영향 기록
- 변경 분류: `데이터 흐름`, `API 계약`
- 생산자: `backend/app/...`의 `function_name`
- 소비자: `/api/...`, `scheduler job`, `frontend/components/...`
- 데이터 경계: 공용 데이터와 `user_id` 기준 개인 데이터의 결합 위치
- API 계약: 인증, 입력, 출력, 오류, 비동기 상태의 변경 내용
- 문서 반영: `docs/...` 또는 `영향 없음 - 구체적인 사유`
- 회귀 테스트: 생산자 단위 테스트와 실제 소비 경로 통합 테스트
- 미해결 위험: 남은 위험과 추적 이슈 번호
- 인수인계: 다음 작업자가 처음 열 파일과 시작할 항목
```

## 5. 생산자-소비자 완결 규칙

변경은 아래 단위가 함께 닫혀야 합니다.

| 변경 대상 | 반드시 확인할 소비자 |
| :--- | :--- |
| API 응답 | 프론트 API 래퍼, 컴포넌트, E2E, API 문서 |
| 공용 캐시 | API 라우터, 스케줄러, 관리자 화면, 시뮬레이터 |
| 사용자별 데이터 | 모든 `user_id` 필터, 공용 캐시 재결합 위치, 2사용자 격리 테스트 |
| 스케줄러 | 수동 실행 경로, 예약 작업, 중복 가드, 상태 표시 |
| DB 모델 | Alembic, 기존 DB 업그레이드, 직렬화, 삭제·보존 정책 |
| 인증·권한 | 백엔드 의존성, 프론트 인터셉터, 미인증·타 사용자 테스트 |
| 운영 절차 | `AGENTS.md`, 관련 가이드, 하네스, CI |
| 중복 기능 로직 | 기존 서비스·훅·계산 모듈, 라우터, 컴포넌트, 스케줄러, 테스트 |

## 6. 문서 반영 기준

- `docs/tasks/YYYY-MM-DD.md`: 모든 작업의 실시간 상태와 인수인계
- `docs/API_STANDARD.md`: 공통 응답·오류 및 장기 API 계약
- `docs/SCHEMA.md`: 모델·컬럼·관계·마이그레이션 계약
- `docs/SYSTEM_MANUAL.md`: 시스템 구성과 전체 데이터 흐름
- `docs/FILTER.md`, `docs/SIGNAL.md`: 스캐너 입력·필터·신호 계약
- `docs/SCANNER_DATA_FLOW.md`: 공용 시장 데이터와 사용자 관심종목의 경계
- `docs/REACT_GUIDE.md`: 신규 React Hook 또는 렌더링 최적화 지식

기존 문서가 실제 구현과 다르면 새 문서만 추가하지 말고 기존 설명도 같은 작업에서 수정합니다.

## 7. 작업 추적 도구 역할

| 도구 | 기록 대상 | 기록하지 않는 것 |
| :--- | :--- | :--- |
| `docs/tasks/YYYY-MM-DD.md` | 그날 수행한 작업, 검증, 승인 상태, 다음 AI 인수인계 | 장기 우선순위 보드 역할 |
| GitHub Issue | 버그·오류·회귀·CI 실패의 증상, 재현, 원인, 수정, 회귀 테스트 | 일반 기능 로드맵과 일일 세부 작업 |
| Linear | 중요한 다일 기능, 운영 인프라, 장기 리팩터링, 상위 안정화 프로젝트 | 당일 체크리스트와 단일 GitHub 버그 복제본 |

상태는 `[ ]` 대기, `[/]` 진행 중, `[R]` 구현·검증 완료/승인 대기, `[x]` 사용자 승인 완료, `[-]` 취소·중복·후속 작업으로 대체를 사용합니다. Linear에는 승인 대기 전용 상태가 없으므로 `[R]` 동안 `In Progress`를 유지하고 `승인 대기` 라벨 또는 댓글을 추가합니다.

## 8. 검증과 종료 절차

1. 수정 Python 파일은 `python -m py_compile`로 검증합니다.
2. 백엔드는 영향 테스트와 전체 `pytest`를 실행합니다.
3. 프론트엔드 변경 시 `npm run lint`, `npx tsc --noEmit`을 실행합니다.
4. 계약 영향 변경은 실제 소비 경로 통합 테스트를 포함합니다.
5. 계산·거래 프로세스 리스크는 `scripts/verify_harness.py`의 `[RISK]` 단계에서 `scripts/check_numeric_invariants.py`와 `scripts/check_process_invariants.py`가 함께 실행되는지 확인합니다.
6. `scripts/verify_harness.py` 전체 검증을 통과합니다.
7. 검증 결과와 미해결 위험을 현황판에 기록합니다.
8. 구현·검증 완료 후 `[R]`로 전환하며, 사용자 승인 전에는 `[x]`, Linear `Done`, GitHub Issue 닫기를 수행하지 않습니다.
9. Git 명령은 사용자의 명시적 트리거가 있을 때만 수행합니다.

## 9. 인수인계 완료 조건

다음 AI가 추가 질문 없이 시작할 수 있어야 합니다.

- 현재 상태가 `[ ]`, `[/]`, `[R]`, `[x]`, `[-]` 중 하나로 명확함
- 변경 이유와 실제 증상이 기록됨
- 수정 파일과 소비 경로가 식별됨
- 실행한 검증과 실패한 검증이 구분됨
- 남은 위험과 다음 첫 단계가 구체적임
- GitHub/Linear 번호와 상태가 실제 외부 상태와 일치함
