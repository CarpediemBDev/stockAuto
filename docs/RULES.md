# 개발 협업 규칙 (StockAuto AI Rules)

이 문서는 AI 어시스턴트(Antigravity)가 본 프로젝트를 진행할 때 **반드시 준수해야 하는 절대 수칙**입니다. 매 대화 시작 시 이 문서를 최우선으로 읽고 스스로를 동기화해야 합니다.

## 1. 협업 및 의사결정 프로세스 (Process & Decision)
*   **비판적 파트너십:** 사용자의 제안에 대해 성능, 보안 리스크, API 차단 위험 등을 선제적으로 분석하여 보고한 뒤 실행합니다. 무비판적인 수용보다는 전문가적인 제언을 우선합니다.
*   **작업 확정 시 선등록 (Task Pre-registration):** 사용자와 작업 내용이 확정되어 **코딩이 시작되는 순간**, AI는 해당 작업 항목을 `docs/task.md`에 `[ ]` 상태로 즉시 추가해야 합니다. 코딩 완료 후 뒤늦게 추가하는 것을 금지합니다.
*   **단계별 점진적 검증:** 대규모 변경 시 [계획 -> 핵심 구현 -> 테스트 -> UI 연결]의 단계를 나누어 진행하며, 각 단계마다 사용자의 방향성 승인을 얻습니다.
*   **승인 기반 업데이트 (Task Completion):** 코딩이 완료되면 사용자에게 보고하고, 사용자의 **최종 승인(Accept)**이 있을 때만 `docs/task.md`의 `[ ]`를 `[x]`로 업데이트합니다. 리젝 시에는 즉시 상태를 롤백합니다.
*   **터미널 역할 분담:** 서버 구동 및 로그 모니터링은 사용자가 전담하며, 그 외 개발 명령어는 AI가 자율적으로 실행하되 파괴적 작업은 사전 승인을 받습니다.
*   **Git 명령어의 완벽한 동결 및 시점 통제 (Absolute Freeze of Git Commands):** AI는 평상시 개발 작업 시 `git status`, `git add`, `git commit`, `git push` 등 **그 어떠한 Git 관련 명령어도 자율적으로 실행해서는 안 됩니다 (조회용 status 포함).** 깃 관련 모든 진단과 실행 명령어는 오직 사용자가 명시적으로 **"커밋하자"** 혹은 **"깃에 올리자"**라고 신호를 보냈을 때만 활성화되어 수행될 수 있습니다. 평소에는 작업 중인 파일들을 순수하게 로컬 작업 트리(Working Tree) 상태로만 유지합니다.
*   **한글 커밋 메시지 표준 (Korean Commit Messages):** 모든 Git 커밋 메시지는 유저가 직관적으로 변경 사항을 추적할 수 있도록 **명확한 한글(Korean)**로 구체적으로 작성합니다. (예: `docs: Git 명령어 완전 동결 및 조회 최소화 규칙 반영`)





## 2. 모듈형 아키텍처 및 문서 체계 (Architecture & Docs)
*   **백엔드 모듈형 레이어드 아키텍처 (Modular Layered):** 백엔드 코드는 무조건 `backend/app/` 아래에 위치하며, 도메인별 폴더(`watchlist/`, `scanner/`, `bot/`, `trades/`, `translations/`)로 격리 관리합니다. 데이터베이스 엔티티, 전역 예외, 설정은 반드시 `app/core/` (`models.py`, `database.py`, `config.py`, `exceptions.py`)로 일원화하고 루트 기준 절대경로 임포트(`from app.core...`)로 통일하여 임포트 꼬임(Circular Import)을 완벽히 차단합니다.
*   **단일 진실 공급원 (SSOT) 및 투명성:** 설계도(`implementation_plan.md`), 현황판(`task.md`), 규칙(`RULES.md`), 전략(`SIGNAL.md`)은 무조건 프로젝트 내부의 **`docs/`** 폴더에서만 관리합니다. **AI는 자신만의 비공개 공간(Artifacts 등)에 정보를 보관하는 것을 엄격히 금지**하며, 모든 계획과 의도는 반드시 프로젝트 폴더 내 문서에 기록하여 투명성을 확보해야 합니다. **모든 코드 변경은 해당 변경 사항을 설명하는 문서(SYSTEM_MANUAL, README 등)의 업데이트와 동시에 이루어져야 합니다. (Doc-Code Sync)**
*   **데이터베이스 마이그레이션 준수 (Alembic SSOT):** 데이터베이스 컬럼 추가/삭제/변경 발생 시, `main.py` 등에 임시 수동 SQL(ALTER TABLE) 코드를 삽입하는 행위를 절대 금지합니다. 무조건 `models.py` 수정 후 `alembic`을 통해 마이그레이션 스크립트를 작성하며, 앱 구동 시 `migrator.py`가 자동으로 이를 감지하여 안전하게 실행(기존 DB는 Stamp, 신규 DB는 Upgrade)하도록 관리합니다.
*   **Vue 마스터용 React 가이드라인 동기화:** 프론트엔드 개발 중 **새로운 React Hook을 도입**하거나, **렌더링 버그/메모리 누수/의존성 충돌 등의 트러블슈팅**이 발생하면, AI는 반드시 그 즉시 **[docs/REACT_GUIDE.md](file:///d:/dev/workspace/stockAuto/docs/REACT_GUIDE.md)** 가이드 문서를 직관적인 비유(Vue ➔ React 매핑 등)와 함께 의무적으로 업데이트해야 합니다.
*   **함수 지향적 모듈화:** 복잡한 상속보다는 독립적인 함수 조합(Composition)을 지향합니다. 모든 신규 함수는 표준 문서화 양식(Google Style / JSDoc)을 준수하여 입출력과 부수 효과를 명시합니다.

## 3. 현업 수준 품질 및 검증 (Quality & Verification)
*   **Production-Grade Standards:** 단순히 돌아가는 코드가 아닌, 실제 프로덕션 환경에서 사용될 수 있는 견고한 코드를 작성합니다. 검증된 오픈소스 레퍼런스를 적극 활용합니다.
*   **프레임워크 표준(Best Practice) 엄격 준수 (No Hacks):** Python(FastAPI), React, Next.js 등 각 언어와 프레임워크가 공식 문서에서 권장하는 **'정석(Standard)' 패턴을 최우선으로 적용**합니다. 린트 에러를 임시방편으로 덮기 위한 꼼수(`setTimeout` 회피 등)나 편법 사용을 엄격히 금지하며, 문제 발생 시 설계 패턴 자체를 프레임워크의 철학(예: React 18+의 렌더링 최적화, 비동기 데이터 펫칭 등)에 맞게 근본적으로 수정해야 합니다.
*   **무결성 검증 필수:** 완료 보고 전 프론트엔드는 `npm run lint` 및 `npx tsc --noEmit`을 실행하여 에러가 0개임을 확인합니다. 추측에 의한 수정을 엄격히 금지합니다.
*   **UI 컴포넌트 주도 개발:** UI 수정 시 공통 컴포넌트와 디자인 시스템을 우선 재사용하여 시각적 일관성을 유지합니다.

## 4. 상세 기술 표준 (Technical Standards)
*   **표준화된 에러 처리:** 모든 API 응답은 일관된 구조(`success`, `data`, `error_message`)를 유지하며, 프론트엔드에서는 공통 훅을 통해 이를 처리합니다.
*   **컨텍스트 기반 로깅:** 에러 발생 시 단순 메시지 대신 당시의 파라미터와 상태(Context)를 포함한 상세 로깅을 수행하여 트레이딩 사고의 원인을 명확히 파악합니다.
*   **보안 및 정적 분석:** 민감 정보의 `.env` 분리, TypeScript `any` 사용 금지, `useEffect` 의존성 배열 관리 등 현업 코딩 체크리스트를 엄격히 준수합니다.
